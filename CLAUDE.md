# CLAUDE.md — правила работы с репозиторием proqnozai-bot

Telegram-бот AI-прогнозов на спорт (Python 3.13, python-telegram-bot 21.5,
Anthropic Claude, SQLite, Railway). Детали архитектуры — в `ARCHITECTURE.md`,
обзор и env-переменные — в `README.md`. Этот файл — правила для работы с кодом.

## Архитектура (кратко)

- Два процесса Railway: `worker` (`main.py` — бот + фоновые задачи +
  `stats_server.py`) и `web` (`dashboard.py` — Flask-дашборд, ходит к
  воркеру только по HTTP).
- Слои снизу вверх: `config` → `db` / `translations` → сервисы
  (`claude_client`, `mostbet`, `football_api`, `security`) → `handlers/` →
  `main`. Импорты только сверху вниз — нижние слои не знают о верхних.

## Команды

```bash
pip install -r requirements.txt -r requirements-dev.txt

pytest tests/               # офлайн-тесты (обязательны перед коммитом)
ruff check .                # линт (постепенная конфигурация)
python -m compileall .      # проверка компиляции

python test_e2e.py          # ручной интеграционный прогон — реальные API,
                            # нужен ANTHROPIC_API_KEY; НЕ запускать в CI
```

## Конвенции кода

- Python с type hints для новых/изменённых функций.
- Комментарии — только о неочевидных ограничениях и «почему», не «что».
- Без лишних абстракций и параллельных реализаций; хендлеры остаются
  «тонкими» — бизнес-логика и парсинг живут в сервисных модулях
  (`mostbet.py`, `football_api.py`, `db.py`), хендлер только склеивает.
- Детерминированные вычисления (суммы, коэффициенты, средние голы, даты,
  сортировки) делаются в Python, а не поручаются LLM.

## Правила, которые нельзя нарушать

### callback_data
Не менять форматы `callback_data` (`fm_sp_`, `fm_lg_`, `fm_mt_`, `watch_`,
`unwatch_`, `fb_`, `repeat_`, `expr_`, `adm_`, `lang_`, `ob_`…) без плана
миграции: старые inline-кнопки остаются в чатах пользователей навсегда и
продолжают слать старый формат. Индексные callback'и (`fm_lg_{i}`) обязаны
строиться и разрешаться одной и той же функцией сортировки
(`_sorted_leagues`), иначе кнопка укажет на чужую лигу.

### Локализация — 7 языков синхронно
Любой новый ключ в `translations.T` добавляется сразу во все 7 языков:
`az, ru, en, tr, kz, uz, ar`. Тест `tests/test_translations.py` проверяет
консистентность ключей — он должен проходить. Локальные словари в хендлерах
(`{"ru": ..., "az": ...}`) тоже покрывают все 7 языков или имеют явный
fallback на `ru`.

### Многострочные строки и `\n`
В переводах и промптах `\n` — это **буквенный эскейп внутри строки**, а не
реальный перенос. При программном редактировании файлов уже ловили
`SyntaxError: unterminated string literal` из-за замены `\n` на настоящий
перенос строки. После правок translations.py всегда прогонять
`python -m compileall .`.

### SQLite-миграции
- Схема создаётся в `db.db_init()` через `CREATE TABLE IF NOT EXISTS`.
- Новые колонки — только аддитивно: `ALTER TABLE ... ADD COLUMN` в списке
  миграций `db_init()` с `try/except sqlite3.OperationalError` (колонка уже
  есть). Не удалять и не переименовывать существующие колонки/таблицы.
- Одноразовые действия — через флаги в `_migrations`
  (`db_flag_done`/`db_flag_mark`).
- Записи в `users` — только через `db_set` с полем из `_ALLOWED_FIELDS`;
  расширять allowlist осознанно, не обходить его сырым SQL.
- Тесты никогда не трогают продовый `bot.db` — только временный файл.

### Переменные окружения
- Секреты только в env; в коде и репозитории — никогда. `.env` в
  `.gitignore`, реальные значения не коммитятся.
- Новая переменная = обновить `.env.example` (плейсхолдер) и таблицу в
  `README.md`.
- `FOOTBALL_KEY` — предпочтительное имя, `FOOTBALL_API_KEY` — legacy
  fallback; не плодить новые алиасы.

### Приватность и логирование
- Не логировать содержимое пользовательских сообщений целиком; допустимы
  усечённые фрагменты в `suspicious.log` для инцидентов (как сейчас,
  `text[:120]`).
- Не логировать значения токенов/ключей ни при каких условиях (длину и
  факт наличия — можно, как в админ-диагностике).
- Пользовательские данные не покидают систему, кроме необходимых вызовов
  Telegram/Anthropic/спорт-API.

### LLM-выдача
- Claude не должен выдумывать травмы, составы, статистику и «факты с
  источниками». Если реальных данных нет — промпт обязан требовать пометку
  «(оценочно)»/«(estimated)». Не ослаблять эти директивы в промптах.
- Реальные коэффициенты и матчи передаются в промпт данными; модель их не
  генерирует.

### Тесты
- Юнит-тесты в `tests/` — строго офлайн: без реальных вызовов Telegram,
  Anthropic, Mostbet и футбольных API. Сеть — только через моки, БД —
  только временная.
- Сетевые интеграционные проверки живут в `test_e2e.py` и запускаются
  вручную.
- Новое поведение = новый тест.
